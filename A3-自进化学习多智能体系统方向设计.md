# A3 自进化学习多智能体系统方向设计

更新时间：2026-06-01（v2：补充推理范式、记忆自进化与微调、MCP 扩展、LLM 网关与任务执行层）

## 1. 方向结论

项目方向建议定为：

> 面向个性化学习资源生成的自进化多智能体系统。

它不是普通 RAG 问答，也不是简单工作流编排，而是一个围绕 Agent 自主性设计的学习系统：Agent 能构建学习画像、拆解学习目标、动态选择检索或工具调用、协同生成多类型学习资源、验证结果质量、记录失败轨迹，并在后续任务中利用长期记忆调整策略。

业务外壳对应第十五届中国软件杯 A3 赛题“基于大模型的个性化资源生成与学习多智能体系统开发”，核心创新落在 Agent 认知架构、记忆反思闭环和知识工程底座。

推荐项目名：

- **ReflexLearn Agent：基于反思记忆的个性化学习多智能体系统**
- **智学云 Agentic Learning：自进化学习资源生成多智能体系统**

## 2. 核心定位

本项目要证明的不是“会调用大模型生成内容”，而是证明以下能力：

1. Agent 有明确状态，不只是无状态聊天。
2. Agent 有推理循环，能根据任务反馈调整策略。
3. Agent 有短期记忆和长期记忆，能复用历史经验。
4. Agent 会失败学习，能把失败轨迹转为下次执行前的策略提示。
5. 多 Agent 不是堆角色，而是有协作协议、验证门、终止条件和冲突处理。
6. RAG 不是简单向量搜索，而是有权限过滤、来源权重、时效权重、冲突识别和混合检索。
7. 数据平台不是为了堆技术栈，而是服务于 Agent 长期记忆和知识持续更新。

## 3. 为什么要保留大数据技术栈

Kafka、Spark、HDFS、Neo4j、Qdrant 不能作为主线无差别堆进去，但应该作为“知识工程与长期记忆底座”的分层能力设计进去。

面试中这些技术栈是亮点，但必须回答清楚每个组件解决什么问题：

| 技术 | 在项目中的合理位置 | 解决的问题 | 面试表达重点 |
|---|---|---|---|
| Qdrant | 长期语义记忆与文档向量库 | 相似经验、知识片段、失败轨迹检索 | RAG 落地、embedding、metadata filter |
| Neo4j | 概念图谱与知识关系层 | 学科知识点依赖、资源引用关系、冲突路径追踪 | 图谱检索、路径推理、GraphRAG |
| Kafka | 知识库增量更新事件流 | 文档变化、爬虫新增、资源更新后触发重建索引 | 流式索引更新、解耦生产与消费 |
| Spark | 离线批处理清洗 | 大规模脏数据清洗、去重、标准化、文本抽取 | 分布式数据处理、数据质量治理 |
| HDFS | 原始语料与中间产物存储 | 保留非结构化原始数据、清洗版本和可追溯数据集 | 数据湖、可复现数据链路 |
| Redis | 热点问题缓存与任务状态 | TTL 缓存、队列状态、短期会话摘要 | 延迟优化、缓存策略 |
| MySQL/PostgreSQL | 业务元数据 | 用户画像、任务记录、资源索引、权限 ACL | 数据建模、SQL 优化、权限控制 |

结论：这些技术栈进入设计，但不抢 Agent 主线。项目叙事应是：

> 为了让 Agent 拥有可持续更新的长期知识和经验记忆，我设计了离线清洗、增量索引、图谱关系和向量检索结合的知识工程底座。

## 4. 总体架构

系统分为五层：

```text
前端交互层
  Next.js / TypeScript / SSE
  学习画像、资源展示、Agent 过程可视化、评测报告

Agent 编排层
  LangGraph StateGraph
  Planner / Router / Executor / Verifier / Critic / Memory Agent
  推理范式：Master 用 Plan-and-Execute，子 Agent 用 ReAct

能力接入层
  LLM 网关（10+ 供应商路由 + HTTP 回退 + 熔断限流计费）
  MCP 工具层（检索 / 图谱 / 资源生成器 / 外部 API）
  taskExecutor 分层并发执行

知识与记忆层
  Qdrant 向量库
  Neo4j 知识图谱
  MySQL/PostgreSQL 元数据
  Redis 缓存与短期状态

数据工程层
  文档上传 / 爬虫采集
  HDFS 原始语料
  Spark 清洗去重
  Kafka 增量更新事件
  混合分块与向量化
```

Agent 编排层是项目核心。能力接入层让模型与工具可插拔。知识与数据工程层是 Agent 长期稳定运行的底座，不应喧宾夺主。

## 5. Agent 角色设计

| Agent | 职责 |
|---|---|
| Master Planner Agent | 理解学习目标，拆分学习任务，决定协作模式 |
| Learner Profile Agent | 通过对话构建学习画像，维护能力、偏好、基础、目标、错误类型等维度 |
| Retrieval Router Agent | 判断当前问题应查向量库、图谱、实时工具还是缓存 |
| Document Resource Agent | 生成讲解文档、知识总结、拓展阅读 |
| Quiz Agent | 生成题目、答案、解析、难度标签 |
| Mindmap Agent | 生成知识结构和学习路径图 |
| Code Practice Agent | 生成代码实操案例和调试任务 |
| Multimodal Resource Agent | 生成多模态脚本、视频分镜、动画说明 |
| Tool Executor Agent | 负责受约束工具调用、参数校验和失败重试 |
| Verifier Agent | 检查资源质量、知识一致性、画像匹配度和格式完整性 |
| Critic / Reflexion Agent | 对失败任务归因，沉淀改进策略和失败轨迹 |
| Memory Manager Agent | 管理短期记忆、长期记忆、摘要压缩和经验检索 |

## 6. Agent 推理范式：ReAct 与 Plan-and-Execute

第 5 节是“有哪些 Agent”，本节是“单个 Agent 内部怎么思考”，第 7 节是“多个 Agent 怎么配合”。三者必须分开讲，这是面试高频追问点。

### 6.1 两种范式对比

| 范式 | 工作原理 | 优点 | 缺点 | 适用场景 |
|---|---|---|---|---|
| **ReAct** | Thought → Action → Observation 循环，每步根据工具/检索反馈再决定下一步 | 灵活、对动态环境适应好、能边走边纠错 | 每步都要 LLM 调用，token 高；长任务易跑偏、迷失目标 | 答疑、检索、代码调试等需要看反馈的任务 |
| **Plan-and-Execute** | Planner 先一次性产出完整计划（步骤列表 / DAG），Executor 逐步执行，必要时 replan | 目标稳定不漂移、步骤可并行、token 可控、过程可观测 | 计划僵化，对中途变化适应差 | 学习目标拆解、端到端资源包生成等结构化任务 |

### 6.2 本项目的混合策略

不二选一，而是分层混合：

- **宏观层用 Plan-and-Execute**：Master Planner 把“学完某知识点”这种目标，一次性拆成资源生成子任务 DAG（讲解文档 / 思维导图 / 题目 / 拓展阅读 / 代码案例），保证目标不漂移、子任务可并行分发给不同 Agent。
- **微观层用 ReAct**：单个子 Agent（答疑、检索路由、代码调试）内部跑 ReAct 循环，因为它们必须根据检索命中、工具返回、报错信息动态调整。
- **触发 replan 的条件**：Verifier 连续失败、检索结果为空、工具不可用、迭代超限 → 回到 Planner 重新规划，而不是在原计划里硬撑下去。

### 6.3 LangGraph 表达

- Plan-and-Execute → `StateGraph` 的有向图 + 条件边（conditional edges），plan 节点产出步骤列表，execute 节点消费，replan 节点形成回边。
- ReAct → 节点内部的 tool-calling 循环，用 `recursion_limit` 兜底防止无限循环。
- 这样既能讲清“为什么用 LangGraph 而不是裸 LangChain Agent”（要的是可控状态机，不是黑盒 AgentExecutor），也对应简历里的 plan-and-execute 设计范式能力。

## 7. 多 Agent 协作模式

系统不只支持一种协作模式，而是由 Master Planner 根据任务类型选择：

| 模式 | 适用场景 | 控制策略 |
|---|---|---|
| 中心化调度 | 复杂学习目标拆解、端到端资源包生成 | 主 Agent 拆任务，子 Agent 执行，Verifier 统一验收 |
| 流水线协作 | 文档解析、分块、抽取、向量化、生成 | A 输出给 B，状态机控制每一步输入输出 |
| 民主协作 | 知识冲突、资料矛盾、策略选择 | 多 Agent 给出观点，Debate/Verifier 识别冲突并裁决 |

防止无限循环的机制：

- LangGraph 状态机限制节点转移。
- 每个任务设置最大迭代次数。
- 每轮 Agent 通信都压缩为摘要。
- 明确定义任务完成标准。
- 检测重复状态和重复工具调用。
- token 预算不足时进入 summary buffer，而不是继续堆上下文。

## 8. 状态与上下文工程

状态设计使用 LangGraph `StateGraph` 和 `TypedDict`，只保存核心变量，避免上下文膨胀。

核心状态字段建议：

```text
task_id
user_id
learning_goal
learner_profile
current_plan
agent_messages_summary
retrieved_chunks
tool_call_trace
generated_resources
verification_result
failure_trace
reflection_notes
iteration_count
token_budget
acl_scope
```

上下文保留策略：

1. 固定保留 system prompt 和任务规则。
2. 保留最近 N 轮关键对话。
3. 旧对话进入 summary buffer。
4. 高价值失败经验进入长期记忆。
5. 检索片段按语义相关性、来源可信度、时效性和权限过滤排序。

## 9. 记忆与 Reflexion 机制

记忆分三类：

| 记忆类型 | 内容 | 存储 |
|---|---|---|
| 短期记忆 | 当前任务状态、最近对话、工具调用结果 | LangGraph state / Redis |
| 长期知识记忆 | 文档知识、知识点关系、历史资源 | Qdrant / Neo4j / MySQL |
| 反思经验记忆 | 失败轨迹、错误原因、修复策略、有效 prompt | Qdrant + metadata |

Reflexion 流程：

```text
任务失败
  -> Verifier 标记失败类型
  -> Critic 分析失败原因
  -> 生成 reflection note
  -> Memory Manager 写入长期经验库
  -> 下次相似任务执行前检索相关失败经验
  -> Planner 调整计划或工具选择
```

这样可以体现 Agent 的自主学习能力：不是一次性生成答案，而是在多轮任务中持续降低重复错误。

## 10. 记忆的自进化与 LoRA 微调

第 9 节解决“记得住、查得到”，本节解决“记忆会沉淀、会压缩、最终能内化进模型”，这是项目“自进化”三个字的真正落点，也是与普通 RAG 项目拉开差距的地方。

### 10.1 记忆的三级演进

| 层级 | 形态 | 机制 | 对应技术 |
|---|---|---|---|
| L1 会话内 | 短期记忆 | summary buffer 递归摘要，旧对话压缩进 context 头部 | LangGraph state / Redis |
| L2 跨会话 | 经验记忆 | Reflexion 写入向量库；高频复用经验晋升为稳定 SOP | Qdrant + metadata |
| L3 沉淀权重 | 模型能力 | 把高质量“轨迹+反思”数据集定期 LoRA 微调，内化为模型本身能力 | HDFS → Spark → LoRA |

### 10.2 借鉴 Claude Code / harness 的上下文工程

- **分层记忆文件 + 索引按需加载**：仿照 CLAUDE.md / MEMORY.md 的做法——长期记忆做成“一条记忆一个文件 + 一个索引”，召回时先看索引摘要判断相关性，再按需加载正文，避免把全部记忆塞进 context。
- **记忆分类型并带元信息**：每条记忆标注类型（用户画像 / 项目约束 / 失败反馈 / 外部引用）和“为什么、如何应用”，让召回时能快速判断是否相关。
- **递归摘要形成层级结构**：旧记忆压缩成更高层摘要，多层级递归，正好对应“通过摘要层级结构对记忆进行递归压缩”的设想。

### 10.3 自进化数据闭环

```text
运行日志/轨迹 (HDFS)
  -> Spark 清洗筛选高质量、可复现的成功+失败轨迹
  -> 构造 LoRA 训练集（结构化决策类节点优先：Planner / Router）
  -> LoRA 微调小模型
  -> LLM-as-a-judge 评测对比微调前后
  -> A/B 灰度上线
```

把数据工程层（第 12 节）和“自进化”真正接上：数据不只是喂 RAG，还能蒸馏回模型权重。

### 10.4 取舍判断（面试要点）

- **该微调**：高频、稳定、结构化的决策模式（如固定的任务拆解套路、路由判断），微调能降低运行时检索开销和 token。
- **该用 RAG**：长尾、易变、需要可追溯来源的知识（如课程内容、最新资料），不该硬塞进权重。
- LoRA 微调是 **L3 扩展项**，不进第一阶段；但能在面试里讲清“何时微调、何时 RAG”，比真的训出一个模型更能体现判断力。

## 11. RAG、图谱与冲突处理

检索不只做向量相似度，而是混合检索：

1. Qdrant 做语义召回。
2. Neo4j 做知识点关系、依赖路径和概念扩展。
3. MySQL/PostgreSQL 做业务元数据、用户画像和 ACL 权限。
4. Redis 做热点问题 TTL 缓存。

排序因素：

- 语义相关性。
- 来源可信度。
- 数据时效性。
- 用户权限范围。
- 历史验证通过率。
- 与当前学习画像的匹配度。

冲突处理：

- 检索片段互相矛盾时，交给 Debate/Verifier Agent。
- 不同 Agent 分别检查来源、时间、定义范围和适用条件。
- 最终输出保留依据，不把冲突简单合并成一个看似确定的答案。

权限控制：

- 文档 chunk 写入向量库时带 `user_id`、`tenant_id`、`course_id`、`visibility` 等 metadata。
- 检索时强制注入 ACL filter。
- 不依赖 prompt 拦截，而是在数据库查询层做物理隔离。

## 12. 数据工程底座设计

数据工程底座采用分层推进。

### L0：第一阶段必须落地

- 文档上传。
- 多格式解析。
- 混合分块（规则保留代码 / 公式 / 表格 + 语义分块）。
- embedding 向量化。
- Qdrant 检索。
- MySQL/PostgreSQL 保存用户、任务、资源、权限元数据。
- Redis 做任务状态和热点缓存。

### L1：第二阶段面试亮点

- Neo4j 建知识图谱，保存知识点依赖、资源引用、错题关系。
- Kafka 监听知识库变化，触发异步清洗、抽取、向量化和图谱更新。
- Spark 对批量语料做去重、清洗、标准化和关键词抽取。
- HDFS 保存原始语料、清洗中间结果和版本化数据集。

### L2：扩展亮点

- 分布式爬虫采集公开学习资料。
- 反爬策略：重试、代理、cookie 池、滑块验证处理。
- 离线索引优化：分块参数评测、embedding 模型对比、索引重建。
- 在线检索优化：rerank、缓存、超时降级、health metrics。

这三层的表达顺序很重要：先讲 Agent 需要长期知识更新，再讲为什么需要数据工程，而不是先抛技术栈。

## 13. 工具调用可靠性

Agent 工具调用必须可控：

- 工具参数用 Pydantic / JSON Schema 强约束。
- LLM 输出走 JSON mode 或 function calling。
- 参数不合法时，把错误返回给 LLM 自修复。
- 高风险操作加入人工确认。
- 工具调用有重试、超时、fallback 和错误日志。
- Tool Executor 记录每次调用的参数、结果、耗时和失败类型。

## 14. MCP 工具与技能扩展

第 13 节解决“工具调用本身可不可靠”，本节解决“工具能力如何标准化、可插拔扩展”，直接对应简历里“MCP tools & skills 开发与集成、为智能体扩展能力边界”。

### 14.1 工具 MCP 化

- 把检索（Qdrant query）、图谱查询（Neo4j）、资源生成器（文档 / 题目 / 思维导图）、外部 API 等封装为 **MCP Server / Tools**，Agent 通过统一协议发现、调用，而不是在代码里硬编码工具。
- 工具 schema 仍用 JSON Schema / Pydantic 定义（与第 13 节一致），MCP 负责工具发现、参数校验、权限边界。
- 价值：新增一个生成器 = 新增一个 MCP tool，不动核心编排逻辑，能力边界可插拔扩展。

### 14.2 Skills 封装可复用工作流

- 把可复用的复杂流程（如“生成一套完整资源包”“出一份带解析的分层测验”）封装成 **skill**，按需加载。
- 借鉴上下文工程：不把所有指令一次性塞进 system prompt，而是按任务类型动态挂载对应 skill，降低上下文占用。

### 14.3 与其它层的解耦

- 工具层 MCP 化、模型层网关化（第 15 节）：换模型不影响工具，加工具不影响模型，两条正交的扩展轴。

## 15. LLM 网关与任务执行层

对应简历里“集成 10+ LLM 供应商与 HTTP 回退策略、增强长链路任务鲁棒性”“taskExecutor 分层并发执行与工具路由”。

### 15.1 统一 LLM 网关

- **多供应商接入**：讯飞星火、OpenAI、Claude、Qwen、DeepSeek、本地 vLLM 等 10+ 供应商统一封装。
- **cost-aware 路由**：按任务类型 / 成本 / 延迟选模型——结构化决策用便宜小模型，复杂推理用强模型，中文场景优先星火。
- **HTTP 回退链**：主模型超时 / 限流 / 报错时，按优先级降级到备用供应商，保证长链路任务不中断。
- **统一治理**：重试、超时、熔断、限流、token 计费、health metrics 全部收敛到网关。

### 15.2 taskExecutor 分层并发

| 层级 | 特征 | 并发策略 |
|---|---|---|
| 编排层 | 任务 DAG，有依赖 | 串行 / 拓扑序，依赖满足才触发 |
| 子任务层 | 多类资源生成，相互独立 | 并行扇出，结果汇聚到 Verifier |
| 工具调用层 | IO 密集（检索、API） | 高并发 + 信号量限流 |

- 实现：FastAPI + asyncio 处理并发，Celery 处理耗时异步任务（如视频生成）。
- 与第 7 节状态机配合：并行节点结果统一汇聚到 Verifier 做验收。

## 16. 评测体系

评测不能只看生成内容是否好看，需要评估 Agent 系统能力。

| 指标 | 含义 |
|---|---|
| 任务完成率 | 多 Agent 是否能完成学习资源包生成 |
| 工具调用成功率 | JSON 参数、工具路由、重试后的成功比例 |
| 检索命中率 | 生成内容是否引用到正确知识片段 |
| 个性化匹配度 | 是否符合学习画像、目标和薄弱点 |
| 资源覆盖度 | 是否覆盖文档、题目、思维导图、拓展阅读、代码/多模态资源 |
| 冲突识别率 | 是否能发现检索片段矛盾 |
| 重复错误下降率 | Reflexion 记忆是否减少同类失败 |
| 范式对比 | ReAct vs Plan-and-Execute 在完成率 / token / 延迟上的差异 |
| 回退命中率 | 主模型失败时 HTTP 回退链的降级成功比例 |
| 微调收益 | LoRA 微调前后在决策节点的 token、延迟、错误率对比 |
| 平均延迟 / P95 延迟 | Agent 链路耗时 |
| 缓存命中率 | 热点问题 TTL 缓存效果 |

可以设计 `LLM-as-a-judge` 评测脚本，把“检索、生成、验证、反思”串成可量化实验，并支持单 Agent / 多 Agent / Reflexion / 微调模型的开关对比。

## 19. 当前关键判断

本项目最重要的取舍是：

> 让大数据技术栈服务 Agent 记忆与知识更新，而不是让 Agent 项目退化成大数据平台展示。

因此，第一阶段必须把 Agent 闭环做清楚；第二阶段再把 Qdrant、Neo4j、Kafka、Spark、HDFS 作为工程增强层逐步接入；LoRA 自进化微调放在最后作为加分项。这样既能保证作品可完成，又能在面试中讲出系统设计深度。

补充取舍原则：

- **范式**：宏观稳定用 Plan-and-Execute，微观灵活用 ReAct，不要二选一。
- **能力扩展**：模型走网关、工具走 MCP，两条正交扩展轴，互不耦合。
- **记忆沉淀**：长尾易变知识用 RAG，高频稳定决策才考虑微调，不要为了用微调而微调。
