## 5. 已识别风险与坏味道

### 5.1 生产安全闭环不完整

当前 API 已要求 Bearer token，但对象级权限、审计、会话撤销、数据库用户体系、上传隔离和 AI Safety Gateway 仍未完成。多用户环境下，认证存在不等于资源访问安全。

### 5.2 视频作业缺少对象归属

`src/reflexlearn/api/routes/video.py` 已接入 `get_current_user`，但 `src/reflexlearn/executor/video_jobs.py` 的 `VideoJob` 当前没有 `tenant_id`、`user_id` 字段。提交和查询只验证登录状态，未验证作业归属。

### 5.3 上传已有 guard，但还不是上传安全网关

`src/reflexlearn/api/upload_guard.py` 已做大小、扩展名、MIME、魔数校验，但还没有隔离区、sha256 记录、病毒扫描占位、内容审核、签名 URL、防盗链和原始文件访问控制。

### 5.4 真实评测证据不足

`docs/08` 的受控基线证明了评测靶场和上限收益，但还不能证明真实 Qdrant/Neo4j/BM25/Reflexion 在生产链路中的稳定收益。需要在同一批 `ablation` case 上补真实链路、LLM-as-a-judge 和人工抽检。

### 5.5 前端仍是演示台

当前前端能力集中在单页工作台，缺少独立路由、会话列表、知识库列表、资源库、评测看板、管理台、用户设置、系统状态和错误恢复。

### 5.6 目录和文件规模触碰项目红线

只读核对结果：

- `tests/unit` 当前 42 个测试文件，超过“每层文件尽量不超过 8 个”的约束。
- `src/reflexlearn/data_engineering`、`src/reflexlearn/rag`、`src/reflexlearn/skills`、`src/reflexlearn/orchestration/nodes` 均超过每层 8 个文件的理想约束。
- `src/reflexlearn/skills/path_plan.py` 当前约 301 行，已经触碰动态语言单文件 300 行约束。

优化建议：

- 后续新增测试前，优先把 `tests/unit` 拆成 `api/`、`data_engineering/`、`eval/`、`executor/`、`memory/`、`orchestration/`、`rag/`、`skills/`。
- 将 `path_plan.py` 拆为 schema/排序策略/LLM 解析/skill 编排等小文件。
- `rag/`、`skills/`、`data_engineering/` 后续新增能力时按子领域分层，不再继续扩大根目录。

## 7. 建议任务包

### 任务包 A：对象级权限与审计

目标：所有知识资源、上传文档、视频任务和敏感操作都绑定 `tenant_id + user_id`，读取和变更时校验归属，并写入审计记录。

建议修改范围：

- `src/reflexlearn/common/auth.py`
- `src/reflexlearn/api/deps.py`
- `src/reflexlearn/api/routes/knowledge.py`
- `src/reflexlearn/api/routes/video.py`
- `src/reflexlearn/executor/video_jobs.py`
- `src/reflexlearn/data_engineering/ingest.py`
- 新增 `src/reflexlearn/common/audit.py`
- 新增或拆分 `tests/unit/api/test_object_access.py`
- 新增或拆分 `tests/unit/common/test_audit.py`
- 更新 `scripts/check_api_security.sh`

验收标准：

- 未登录访问 chat / knowledge / video 继续返回 401。
- 登录用户不能读取其他用户或其他租户的视频作业。
- 视频作业持久化包含 `tenant_id`、`user_id`、`created_at`。
- 上传文档和视频任务的关键操作写入结构化审计日志。
- 审计日志不记录 token、密钥、原始密码和大段文件内容。

验证命令：

```bash
bash scripts/test_unit.sh tests/unit/api tests/unit/common -q
bash scripts/test_unit.sh
bash scripts/check_api_security.sh 8000
```

### 任务包 B：AI Safety Gateway

目标：建立统一 AI 输入/输出安全边界，不把 prompt 约束当成安全机制。

建议修改范围：

- 新增 `src/reflexlearn/safety/`
- `src/reflexlearn/api/routes/chat.py`
- `src/reflexlearn/api/routes/knowledge.py`
- `src/reflexlearn/orchestration/graph.py`
- `src/reflexlearn/llm_gateway/gateway.py`
- 新增或拆分 `tests/unit/safety/`
- 更新 `scripts/check_api_security.sh`

最小能力：

- 输入审核：拦截明显越权、密钥索取、提示词注入、恶意文件说明。
- 输出审核：对敏感信息、内部路径、密钥形态、越权解释做过滤或降级。
- 事件审计：记录命中规则、用户、租户、请求类型、trace_id。
- 降级策略：命中高风险时返回明确错误，不继续进入 LLM 或工具链。

验收标准：

- 提示模型泄露系统提示词的请求被阻断或脱敏。
- 上传文件标题/内容中的明显注入片段不会直接进入工具调用。
- LLM 输出中疑似密钥形态内容被脱敏。
- 规则命中写入审计日志。

验证命令：

```bash
bash scripts/test_unit.sh tests/unit/safety -q
bash scripts/test_unit.sh
bash scripts/check_api_security.sh 8000
```

### 任务包 C：上传隔离与签名 URL

目标：把“基础上传 guard”升级为“上传安全网关”，避免原始文件直接进入可信区。

建议修改范围：

- `src/reflexlearn/api/upload_guard.py`
- `src/reflexlearn/api/routes/knowledge.py`
- `src/reflexlearn/data_engineering/storage.py`
- `src/reflexlearn/data_engineering/ingest.py`
- `scripts/check_api_integrations.sh`
- 新增或拆分 `tests/unit/data_engineering/test_upload_isolation.py`

最小能力：

- 原始文件先进入 MinIO 隔离区或本地隔离目录。
- 记录 sha256、大小、MIME、扩展名、上传者、租户、扫描状态。
- 病毒扫描先做可替换占位实现，状态包括 `pending`、`clean`、`rejected`、`unavailable`。
- 只有 `clean` 文件进入解析和入库链路。
- 下载或预览使用短期签名 URL，不暴露真实对象路径。

验收标准：

- 非法类型、空文件、超大文件继续被拒绝。
- 同一文件可通过 sha256 追踪。
- 扫描不可用时明确 degraded，不假装 clean。
- 签名 URL 过期后无法访问。

验证命令：

```bash
bash scripts/test_unit.sh tests/unit/data_engineering -q
bash scripts/test_unit.sh
bash scripts/check_api_integrations.sh 8000
```

### 任务包 D：真实评测证据链

目标：把当前受控上限基线扩展为比赛/答辩可信证据。

建议修改范围：

- `src/reflexlearn/eval/`
- `scripts/run_eval.py`
- `scripts/run_eval.sh`
- `docs/08-评测报告与消融结果.md`
- 新增或拆分 `tests/unit/eval/`
- 前端看板阶段再接 `frontend/app/eval/`

执行顺序：

1. 固定假 Qdrant / 假 MemoryManager 注入主链路消融，避免外部服务波动。
2. 在完整中间件环境下放开真实 RAG / Reflexion，跑同一批 `ablation` case。
3. 有 LLM key 后运行 LLM-as-a-judge。
4. 抽样人工复核规则 Judge 与 LLM Judge 的一致性。
5. 将 JSON/Markdown 报告沉淀到 `docs/08` 或新版本评测报告。

验收标准：

- 同一批 case 能产出 no_rag / controlled_rag / real_rag 对比。
- 同一批 case 能产出 no_reflexion / controlled_reflexion / real_reflexion 对比。
- 报告明确区分 smoke、受控基线、真实链路、人工抽检。
- 不夸大规则 judge 结论。

验证命令：

```bash
bash scripts/run_eval.sh --compare --tags ablation,rag_required --max-cases 1 --timeout 12
bash scripts/run_eval.sh --compare --tags ablation,reflexion_required --max-cases 1 --timeout 12
bash scripts/test_unit.sh tests/unit/eval -q
```

## 9. 暂缓事项

以下事项不建议马上抢主线：

- LoRA 微调：没有稳定训练数据和 GPU 前，收益不确定。
- 复杂多进程 Agent / ACP / MCP 协议化协作：当前更紧急的是安全、评测和产品结构。
- 爬虫与大规模数据采集：上传安全与知识库管理未稳前，不宜扩大数据入口。
- 完整 SaaS 计费、组织管理、复杂 RBAC：先完成对象归属、审计和最小角色校验。

## 10. 下一次接手建议

下一次开发最建议从“任务包 A：对象级权限与审计”开始，原因：

- 它直接覆盖 `docs/10` 标出的最高风险。
- 它能顺带修复视频作业对象归属缺口。
- 它是 AI Safety、上传隔离、前端资源库和评测看板的共同基础。
- 它的范围可控，适合继续沿用已有 P0 执行计划的 TDD 风格。

开始前建议先新建更细的执行计划：

```text
discuss/P0-对象级权限与审计执行计划.md
```

该执行计划应包含精确文件清单、测试先行步骤、每步验证命令和执行回报模板。
