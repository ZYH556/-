## 1. 已锁定的全局约束（2026-06-01 确认）

| 维度 | 决策 | 影响 |
|---|---|---|
| 目标定位 | 真打中国软件杯 A3（功能完整度对齐评分项） | P0 必须覆盖赛题全部硬指标 |
| 大数据落地 | 全套真搭集群（docker-compose） | Kafka / Spark / HDFS / Neo4j 全部真跑 |
| 模型 / 算力 | 国内 API + 海外 API + 本地模型 + GPU | LLM 网关多供应商、本地 embedding、LoRA 全可行 |
| 后端栈 | 全 Python / FastAPI | Agent 编排与后端统一，Celery 异步 |

**简历价值与比赛目标在此是统一的**：比赛的「创新性 + 技术实现质量」评分项，正好就是简历要展示的 Agent 架构深度。因此 P0 保比赛功能基线，P1 做创新深度（同时拿比赛创新分 + 简历亮点）。

## 2. 项目参数（2026-06-01 确认）

| 参数 | 取值 | 对规划的影响 |
|---|---|---|
| 开发周期 | 不设硬 deadline，给足周期 | 里程碑按相对阶段推进、不压缩；范围仍按 P0→P1→P2 顺序保证随时可交付 |
| 团队 | 2 人：1 人主开发（vibecoding），1 人负责 PPT / 文档 | **核心开发并行度 = 1**；规划须 AI-friendly、模块解耦、接口先行 |
| 课程领域 | 机器学习 | 知识库、测试数据、题库、代码案例统一围绕 ML 组织 |
| 投入强度 | 接近全职 | — |

> 关键推论：**实际写代码的只有 1 人 + AI 辅助**。因此架构设计第一原则是「模块边界清晰 + 接口契约先行 + 优先成熟库少自研」，让每个模块都能被 AI 独立、低上下文地实现与测试。

## 3. 范围分层（真打比赛 → 先保 P0）

### P0 · 比赛硬指标（必达，对齐评分表）
- 对话式学习画像，**≥6 维度**，支持「随学随新」动态更新
- 多智能体协同，**≥5 个专业 Agent**，有明确协作机制与可视化
- **≥5 种资源生成**：讲解文档 / 思维导图 / 练习题 / 拓展阅读 / 多模态视频或动画 / 代码案例
- 个性化学习路径规划 + 资源精准推送
- 流式输出 + 前端多模态卡片化（Markdown / 导图 / 视频 / 代码高亮）
- docker 一键部署 + 配套文档（开发 / 测试 / 部署说明书）+ AI Coding 工具标注

### P1 · 创新深度（拉开差距 + 简历核心）
- ReAct / Plan-and-Execute 混合推理 + replan
- Reflexion 失败学习闭环（重复错误下降率可量化）
- 三级记忆 + 递归摘要 + summary buffer
- 混合 RAG（Qdrant 语义 + Neo4j 图谱 + 关键词）+ rerank + ACL 物理隔离 + 冲突辩论
- LLM-as-a-judge 评测 + 单 / 多 Agent / Reflexion 消融对比
- Agent 协作过程可视化（赛题明确加分点）
- 学习画像可解释（展示「为什么推荐这个资源」）
- **Skills-Agent 两层架构 + Harness Engineering 治理**（约束三层校验 / 失败分层兜底 / 熵管理 / Skill 限次）

### P2 · 工程扩展（行有余力 / 决赛加强 / 简历额外亮点）
- 全套大数据增量链路（Kafka → Spark → HDFS → 重建索引）
- 分布式爬虫 + 反爬（cookie 池 / 重试 / 滑块）
- LoRA 自进化微调闭环
- 智能辅导、学习效果评估（赛题可选加分项）

> **砍单顺序（时间不够时从下往上砍）**：LoRA 微调 → 爬虫 → 大数据实时增量链路（保留离线批处理演示）→ 部分多模态降级为图文。**P0 永不砍。**

## 4. 技术选型总表

> 原则：能用成熟开源就不自研；每个组件都能回答「解决什么问题」。标 🔬 的是面试可讲的前沿点。

### 4.1 前端与交互
| 选型 | 用途 | 理由 / 前沿点 |
|---|---|---|
| Next.js 14 (App Router) + TypeScript | 前端框架 | 赛题鼓励、SSR / 流式友好 |
| Tailwind + shadcn/ui | UI 组件 | 快速出美观界面（评分项：界面美观） |
| SSE（主）/ WebSocket | 流式传输 | 流式输出避免空白等待（非功能需求） |
| react-markdown + KaTeX | 文档 / 公式渲染 | |
| Markmap | 思维导图 | Markdown → 可交互导图 |
| Monaco / Shiki | 代码高亮 | 代码案例展示 |
| Agent 时间线组件 | 协作可视化 | 🔬 多智能体过程可视化（加分项） |

### 4.2 Agent 编排
| 选型 | 用途 | 理由 / 前沿点 |
|---|---|---|
| LangGraph (StateGraph) | 编排核心 | 🔬 可控状态机 > 黑盒 AgentExecutor |
| checkpointer (Postgres/Redis) | 状态持久化 | 🔬 断点续跑、时间旅行调试 |
| interrupt | 人机协同 | 🔬 高风险操作人工确认 (human-in-the-loop) |
| subgraph | 子流程封装 | 各资源生成 Agent 模块化复用 |
| PaE + ReAct 混合 | 推理范式 | 详见 02 文档 |
| Skills-Agent 两层 | 能力/决策分层 | 🔬 原子 Skill 复用 + 限次 + 缓存 + 监控（借鉴 Agent 包装稿） |
| Harness Engineering | 运行护栏 | 🔬 YAML 约束 + 三层校验 + 失败兜底 + 安全升级（呼应「参考 harness」） |

### 4.3 LLM 与生成
| 选型 | 用途 | 理由 / 前沿点 |
|---|---|---|
| LiteLLM | LLM 网关 | 一套接口统一 10+ 供应商 + 回退 + 计费，省自研 |
| DeepSeek-V3 / 星火X1.5（推理）、星火/Qwen（生成）、Claude（强模型兜底）、本地 Qwen2.5(vLLM) | 模型路由 | 🔬 cost-aware routing：按任务 / 成本 / 延迟选模型 |
| bge-m3（本地） | embedding | 中文好、开源、稠密+稀疏一体 |
| bge-reranker（本地） | 重排 | 🔬 rerank 提升检索精度 |
| SeeDance（赛题指定）/ 可灵 / 即梦 | 视频生成 | 赛题明确点名 SeeDance |
| 文生图 API / SD | 配图 | 文档与 PPT 配图 |

### 4.4 存储与检索
| 选型 | 用途 | 理由 / 前沿点 |
|---|---|---|
| Qdrant (docker) | 向量库 | metadata filter 强，ACL 友好 |
| Neo4j (docker) | 知识图谱 | 🔬 GraphRAG、知识点依赖路径 |
| PostgreSQL | 关系库 | 画像 / 任务 / 资源 / ACL；JSONB 灵活 |
| Redis | 缓存 / 短期 / broker | TTL 缓存、会话状态、Celery broker |
| Mem0 | 记忆管理层 | 🔬 记忆抽取 / 更新 / 合并 / 去重 / 语义召回；Qdrant 作其向量底座 |

### 4.5 数据工程
| 选型 | 用途 | 理由 / 前沿点 |
|---|---|---|
| Kafka (docker) | 事件流 | 知识库增量更新解耦生产/消费 |
| Spark (docker) | 批处理 | 大规模清洗、去重、标准化 |
| HDFS (docker) 或 MinIO | 原始语料 / 数据湖 | MinIO 更轻、S3 兼容（二选一，倾向 HDFS 贴赛题叙事） |
| 规则(代码/公式/表格)+语义 混合分块 | 分块 | 🔬 结构感知分块 |
| Contextual Retrieval | 检索增强 | 🔬 给 chunk 加上下文摘要再 embed（Anthropic），显著降召回失败率 |

### 4.6 执行与异步
| 选型 | 用途 | 理由 |
|---|---|---|
| FastAPI + asyncio | API / 并发 | 全 Python 主栈 |
| Celery + Redis | 异步重任务 | 视频生成等耗时任务 |
| taskExecutor 分层并发 | 任务调度 | 详见 05 文档 |

### 4.7 评测与可观测
| 选型 | 用途 | 理由 / 前沿点 |
|---|---|---|
| 自研 eval harness + LLM-as-a-judge | 评测 | 🔬 0–1 打分、消融对比 |
| RAGAS（可选） | RAG 评测 | 检索 / 生成质量指标 |
| LangSmith 或 OTel + Prometheus + Grafana | trace / 监控 | 🔬 全链路 trace、health metrics |

### 4.8 微调与部署
| 选型 | 用途 | 理由 / 前沿点 |
|---|---|---|
| PEFT / LoRA + Unsloth | 微调 | 🔬 Unsloth 加速、省显存 |
| vLLM | 本地推理 / 微调模型部署 | 高吞吐 |
| docker-compose | 一键部署 | 比赛部署要求 |

## 7. 关键风险与预案

| 风险 | 影响 | 预案 |
|---|---|---|
| 范围三重最大化，做不完 | 高 | 严格 P0 优先；里程碑可独立演示；砍单顺序明确 |
| 多模态视频生成慢 / 不稳 / 收费 | 中高 | 异步生成 + 预生成缓存 + 降级图文/数字人；SeeDance 不可用时备可灵/即梦 |
| 大数据集群调试吃时间 | 中 | docker-compose 模板化 + 健康检查脚本；非阻塞主链路；先离线后实时 |
| LoRA 收益不确定 | 中 | 作为 P2；先 RAG，eval 证明有收益再上；Unsloth 加速 |
| 多供应商 API 限流 / 波动 | 中 | LLM 网关回退链 + 本地 vLLM 兜底 |
| 多人协作代码冲突 | 中 | 模块边界 + 接口先定（02 / 05 文档先行）；Python 包结构清晰 |
| 评测变玄学 | 中 | 评测先行（M1 就搭最小 eval），Evaluation-Driven |
