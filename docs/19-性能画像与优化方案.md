# 19 · 主链路性能画像与优化方案（P0 性能线）

更新时间：2026-06-10 · 性质：性能专项分析与派工预案 · 背景：用户拍板「性能优化」为下一主攻方向

> **问题**：带元认知的真实生成全链路 39–90s（`metacognition_real_on latency_ms=38982`，且这是 Judge 限 1 资源、关 RAG 的瘦身条件）；商用对标产品首响应为秒级。
> **方法**：本文基于 2026-06-10 代码静态画像（`orchestration/graph.py` / `llm_gateway/gateway.py` / `nodes/core/generator.py` / `api/routes/chat.py`），定位串行链与调用模式，给出按 ROI 排序的优化卡。落地前应先用 P1-3 的耗时观测验证画像。

---

## 1. 耗时画像：一次 /chat 请求的关键路径

默认生产配置（`enable_llm_profile/planner/quality_check/generation` 全开，central 模式，3 资源计划）：

```
串行 LLM 往返（每次经中转站 5–15s）：
profile(1次LLM) → planner(1次LLM) → fan-out 并行×N资源 ┐
  每资源内部串行: RAG检索 → 生成(1次LLM) → 质检(1次LLM) ┘ 关键路径取最慢资源 = 2次LLM
→ path_plan(1次LLM) → PERSIST(多轮溢出时+1次摘要LLM，阻塞 done 帧)

关键路径 = 5 次串行 LLM 往返 ≈ 25–75s   ← 与 39–90s 观测吻合
```

叠加因素：

| # | 因素 | 量级 | 出处 |
|---|---|---|---|
| 1 | **全部 LLM 调用非流式**：`litellm.acompletion` 与 httpx 一次性 POST，资源整块生成完才 emit；SSE 只是「整块推送」不是 token 流 | 首个实质内容要等 3 次往返（~15–45s）才出现 | `gateway.py:82,140-146` |
| 2 | **httpx.AsyncClient 每次调用新建**：无连接池，每次 LLM 调用重做 TCP+TLS 握手 | +0.3–1s × 每次调用 | `gateway.py:140` |
| 3 | **质检每资源 1 次完整 LLM**：判断类任务用与生成同档模型 | 每资源 +5–15s | `generator.py:68` |
| 4 | **PERSIST 阻塞 done 帧**：递归摘要在 `run_session` generator 耗尽前执行，chat.py 循环退出后才发 done | 多轮溢出时 +3–10s 连接挂住 | `graph.py:239-297` |
| 5 | bge-large-zh 懒加载（2GB） | 首请求一次性 +10–30s | `common/embedding.py` |
| 6 | 中转站本身（timicc /responses）排队与慢响应 | 不可控变量，60s 评测窗口曾不稳定 | docs/08 §7 |

## 2. 优化卡（按 ROI 排序）

### PERF-A · Token 级流式生成 🥇（体感质变：首内容 30-45s → 2-5s）

对标 Claude Code 等智能体的核心体验差异：不是总时长短，而是**即时反馈**。

- `LLMGateway` 新增 `complete_stream()`：litellm `acompletion(stream=True)` / httpx `client.stream()`，逐 chunk yield；保留现有 `complete()` 零回归。
- `doc_gen` 等主资源 skill 走流式；`generate_resource` 把增量经队列上抛；`chat.py` 新增 `resource_delta` SSE 帧（带 `task_id` 区分 fan-out 多路）。
- 前端 ResourceCard 增量渲染（SSE 解析已就绪）。
- 降级铁律：流式异常 → 回退一次性 `complete()`；中转站不支持 stream → 自动回退。
- 风险：fan-out 并行流的多路复用顺序；schema 类调用（planner/质检）不流式，保持原样。

### PERF-B · 砍串行前置与 cheap 路由 🥈（总时长 -40%）

- **profile + planner 合并**为一次 LLM 调用（一个 JSON 同时产出画像增量与资源计划），或 profile 降为规则/cheap 档——Claude Code 哲学：没有固定前置 pipeline，让主模型直接开干。
- **质检/path_plan/judge 等判断类任务走 cheap 档**：`CHEAP_TASK_TYPES` 机制已存在（summary 已用），把 `judgment`/`quality`/`path_plan` 加入并配 `*_model` 覆盖项。
- 单独验证质检 cheap 化后 gate 通过率不回归（eval ablation 切片守门）。

### PERF-C · 连接复用 + PERSIST 后台化 + 超时分级

- 模块级 httpx.AsyncClient 单例（keep-alive 连接池），省每调用 TLS 握手。
- PERSIST（递归摘要/promote/autogrow）用 `asyncio.create_task` 后台化，done 帧不再等待；注意后台任务异常吞掉并记日志（已有降级惯例）。
- 超时分级：profile/planner/质检 10–12s，生成类 60s（配流式后超时语义改为「首 chunk 超时」）。

### PERF-D · 预热与基础设施

- API 启动时后台预热 bge 模型（消除首请求 10–30s）。
- Grafana 加 LLM p95 延迟面板（`reflexlearn_llm_*` 指标已埋好），先量化再优化中转站选型；必要时直连 DeepSeek/Qwen 官方替代慢中转。

## 3. 验收口径

- 指标：**TTFC**（time-to-first-content，首个资源增量帧）与 **总时长**；用 `check_metacognition_perf.sh` 同源方法新增 `check_chat_perf.sh` 采集。
- 目标：TTFC < 5s；3 资源总时长 < 35s（中转站正常时）。
- 质量守门：`run_eval.sh --compare --tags ablation` 分数无回归；477 单测零回归（W3-E 后基线）。
- 诚实边界：中转站排队是外部变量，报告需分别记录「网关耗时」与「端到端耗时」。

## 4. 实施顺序建议

PERF-B（纯后端、风险低、立竿见影）→ PERF-C → PERF-A（动前后端契约，单独一轮）→ PERF-D。每卡独立可验收，沿用降级铁律与零回归纪律。
