# 19 · 主链路性能画像与优化方案（P0 性能线）

更新时间：2026-06-10 · 性质：性能专项分析与派工预案 · 背景：用户拍板「性能优化」为下一主攻方向

> **问题**：带元认知的真实生成全链路 39–90s（`metacognition_real_on latency_ms=38982`，且这是 Judge 限 1 资源、关 RAG 的瘦身条件）；商用对标产品首响应为秒级。
> **方法**：本文基于 2026-06-10 代码静态画像（`orchestration/graph.py` / `llm_gateway/gateway.py` / `nodes/core/generator.py` / `api/routes/chat.py`），定位串行链与调用模式，给出按 ROI 排序的优化卡。落地前应先用 P1-3 的耗时观测验证画像。

## 1. 耗时画像：一次 /chat 请求的关键路径

默认生产配置（`enable_llm_profile/planner/quality_check/generation` 全开，central 模式，3 资源计划）：

```
串行 LLM 往返（每次经中转站 5–15s）：
profile(1次LLM) → planner(1次LLM) → fan-out 并行×N资源 ┐
  每资源内部串行: RAG检索 → 生成(1次LLM) → 质检(1次LLM) ┘ 关键路径取最慢资源 = 2次LLM
→ path_plan(1次LLM) → PERSIST(多轮溢出时+1次摘要LLM，阻塞 done 帧)

关键路径 = 5 次串行 LLM 往返 ≈ 25–75s   ← 与 39–90s 观测吻合
```

叠加因素：

| # | 因素 | 量级 | 出处 |
|---|---|---|---|
| 1 | **全部 LLM 调用非流式**：`litellm.acompletion` 与 httpx 一次性 POST，资源整块生成完才 emit；SSE 只是「整块推送」不是 token 流 | 首个实质内容要等 3 次往返（~15–45s）才出现 | `gateway.py:82,140-146` |
| 2 | **httpx.AsyncClient 每次调用新建**：无连接池，每次 LLM 调用重做 TCP+TLS 握手 | +0.3–1s × 每次调用 | `gateway.py:140` |
| 3 | **质检每资源 1 次完整 LLM**：判断类任务用与生成同档模型 | 每资源 +5–15s | `generator.py:68` |
| 4 | **PERSIST 阻塞 done 帧**：递归摘要在 `run_session` generator 耗尽前执行，chat.py 循环退出后才发 done | 多轮溢出时 +3–10s 连接挂住 | `graph.py:239-297` |
| 5 | bge-large-zh 懒加载（2GB） | 首请求一次性 +10–30s | `common/embedding.py` |
| 6 | 中转站本身（timicc /responses）排队与慢响应 | 不可控变量，60s 评测窗口曾不稳定 | docs/08 §7 |

## 2. 优化卡（按 ROI 排序）

### PERF-A · Token 级流式生成 🥇（体感质变：首内容 30-45s → 2-5s）

对标 Claude Code 等智能体的核心体验差异：不是总时长短，而是**即时反馈**。

- `LLMGateway` 新增 `complete_stream()`：litellm `acompletion(stream=True)` / httpx `client.stream()`，逐 chunk yield；保留现有 `complete()` 零回归。
- `doc_gen` 等主资源 skill 走流式；`generate_resource` 把增量经队列上抛；`chat.py` 新增 `resource_delta` SSE 帧（带 `task_id` 区分 fan-out 多路）。
- 前端 ResourceCard 增量渲染（SSE 解析已就绪）。
- 降级铁律：流式异常 → 回退一次性 `complete()`；中转站不支持 stream → 自动回退。
- 风险：fan-out 并行流的多路复用顺序；schema 类调用（planner/质检）不流式，保持原样。

### PERF-B · 砍串行前置与 cheap 路由 🥈（总时长 -40%）

- **profile + planner 合并**为一次 LLM 调用（一个 JSON 同时产出画像增量与资源计划），或 profile 降为规则/cheap 档——Claude Code 哲学：没有固定前置 pipeline，让主模型直接开干。
- **质检/path_plan/judge 等判断类任务走 cheap 档**：`CHEAP_TASK_TYPES` 机制已存在（summary 已用），把 `judgment`/`quality`/`path_plan` 加入并配 `*_model` 覆盖项。
- 单独验证质检 cheap 化后 gate 通过率不回归（eval ablation 切片守门）。

### PERF-C · 连接复用 + PERSIST 后台化 + 超时分级

- 模块级 httpx.AsyncClient 单例（keep-alive 连接池），省每调用 TLS 握手。
- PERSIST（递归摘要/promote/autogrow）用 `asyncio.create_task` 后台化，done 帧不再等待；注意后台任务异常吞掉并记日志（已有降级惯例）。
- 超时分级：profile/planner/质检 10–12s，生成类 60s（配流式后超时语义改为「首 chunk 超时」）。

### PERF-D · 预热与基础设施

- API 启动时后台预热 bge 模型（消除首请求 10–30s）。
- Grafana 加 LLM p95 延迟面板（`reflexlearn_llm_*` 指标已埋好），先量化再优化中转站选型；必要时直连 DeepSeek/Qwen 官方替代慢中转。

## 3. 验收口径

- 指标：**TTFC**（time-to-first-content，首个资源增量帧）与 **总时长**；用 `check_metacognition_perf.sh` 同源方法新增 `check_chat_perf.sh` 采集。
- 目标：TTFC < 5s；3 资源总时长 < 35s（中转站正常时）。
- 质量守门：`run_eval.sh --compare --tags ablation` 分数无回归；477 单测零回归（W3-E 后基线）。
- 诚实边界：中转站排队是外部变量，报告需分别记录「网关耗时」与「端到端耗时」。

## 4. 实施顺序建议

PERF-B（纯后端、风险低、立竿见影）→ PERF-C → PERF-A（动前后端契约，单独一轮）→ PERF-D。每卡独立可验收，沿用降级铁律与零回归纪律。

## 5. 实测基线（2026-06-14 · `scripts/check_chat_perf.sh`）

首次用 `check_chat_perf.sh` 采集真实 `/api/chat` SSE（admin、"线性回归从入门到精通"、中转站当时可连但慢）：

| 指标 | 值 | 含义 |
|---|---|---|
| first_signal | **0.2s** | session/agent_step 过程帧立即返回（体感"已开始"） |
| TTFC（首 resource_card） | **186s** | 首个**实质内容**几乎和结束同时——体验灾难 |
| total（done） | **187s** | 端到端 |
| llm_calls_this_request | **15** | 本次外呼 15 次（profile1+planner1+6资源×(gen+质检)12+path_plan1=15，与画像吻合） |

**核心结论**：187s / 15 次 ≈ **12.5s/次且近乎完全串行**——`first_signal` 快但 TTFC≈total，证明 ①全程非流式（资源整块生成完才 emit）②**fan-out 6 资源疑似未真并行**（若并行，关键路径应≈单资源链 ~25s，而非 187s）。

**另一环境观测**（中转站 SYN 黑洞、ConnectError）：每资源 connect 等满 30s 总超时才降级，5 资源串行 220s+。本轮已修为 **connect 超时分级**（`llm_connect_timeout_s=5s` 独立于 read 30s），SYN 黑洞时单次 5s 快速降级；中转站可连但慢时该项不改善（瓶颈在 read+串行）。

**下一轮 PERF 优先级（基于基线修正）**：
1. **查 fan-out 是否真并行**（`graph.astream` 对 generate_resource fan-out 的实际执行；若串行，并行化是最大 ROI，total 187s→~30s）。注意：动 LangGraph 主链路需谨慎、单独一轮。
2. **PERF-A 流式骨架**：TTFC 186s→2-5s 的唯一解；gateway `complete_stream()` 先建能力 + 单测，再接 generator/chat（动契约，单独一轮）。
3. 验收锚点：本基线 TTFC≈total（串行非流式）；并行化目标 total≈max单资源链；流式目标 TTFC<5s。诚实分离"网关 LLM 耗时"与"端到端"（`llm_calls_this_request` 已采集）。

### 5.1 PERF-A 进度（2026-06-14）

- ✅ **第 1 步 · 网关流式能力**（FS-10）：`LLMGateway.complete_stream()` 逐 `StreamChunk` yield 增量、末帧带聚合 `Completion`；compat=`httpx.stream()`、litellm=`acompletion(stream=True)`；`openai_compat.stream_payload/stream_delta` 支持 chat + responses 两种 wire。降级铁律完整（未产出→回退 `complete()`；已产出断流→收尾不重复回退；无凭证/关闭抛错；schema 不流式）。`complete()` 零改动。`test_gateway.py` +8（runtime 28 passed）。

## 6. fan-out 串行诊断结论（2026-06-14 · 只加诊断、未改主链路）

按"先确认根因再改"原则，用 `check_chat_perf.sh`（增强版：LLM 按 task_type/status 差分 + resource_card 时间分布）+ `logs/api.log` 的 httpx 外呼时间戳实测：

**铁证（fan-out 确为串行，非并行）**：单次请求 15 次 LLM 外呼时间戳跨 51:35→54:44（**189s**），**逐个间隔 20–39s**、成对紧邻（每资源 generation+verification 两次相邻）、资源间大间隔。若 6 资源真并行，6 个 generation 的 httpx POST 应挤在开头数秒——实测逐个推进，证明 `generate_resource` 串行执行（一个完全结束才下一个）。

**另两项确认**：
- **完全非流式**：12 个 `resource_card` 帧全挤在末尾 0.03s 内（首=末=191.94s），生成阶段零增量、全憋到 assemble 一次性 emit → TTFC≈total。
- **中转站当前 502**：`timicc.com/responses HTTP/1.1 502`，本次 `verification/error×6 + planning/error×2 + profiling/error×1 + generation 5ok/1err`；502 路径每次 hang 20–39s，叠加串行 = 189s。connect 超时分级（PERF-C）对 502（能连上）不改善，仅对 SYN 黑洞有效。

**已排除**：`harness_guard` 无锁（仅 `await node_fn`）；`graph.astream` 无 max_concurrency 限制；`dispatch_route` 正确返回 `fan_out` 的 `Send` 列表（LangGraph 层声明正确）。

**根因候选（下一轮按序验证，不依赖中转站）**：
1. **最小复现隔离 LangGraph Send 并行性**：写单测 fan_out 2 个 mock 资源（mock LLM 各 `asyncio.sleep(2)`），断言总时长≈2s（并行）还是≈4s（串行）。这能在不碰中转站/不改主链路下定位是 LangGraph 调度问题还是节点内串行资源。
2. 若 LangGraph 确并行 → 查 `generate_resource` 内串行资源：RAG retrieve 的 embedding 单例 / `KeywordIndex` 锁（见 memory 陷阱 #9 同步锁阻塞事件循环）贯穿 6 资源。
3. 仅在最小复现确认根因后改并行，不先做 LangGraph 大重构。

**诊断工具备注**：`enable_generator_diagnostics` 经 `start_api.sh` 后台启动时 env 未透传到 uvicorn 子进程（前台 `python -c` 验证 env→settings 链路正常）；本轮改用 httpx 外呼时间戳（更直接，每次真实外呼），结论等价更强。下次需 generator_diag 走 `.env` 临时行或前台启动。

## 7. fan-out 串行根因确认 + 修复（2026-06-15 · FS-12）

按 §6「先确认根因再改」，用最小复现逐层隔离（不依赖中转站、不改主链路）：

| 隔离层 | 实验 | 结论 |
|---|---|---|
| LangGraph `Send` 扇出 | 纯 `asyncio.sleep` worker × 4，astream | 4×1s=**1.03s → 并行**（调度层无罪） |
| LLM 网关 | 真 `LLMGateway`，httpx 替身 sleep 1s × 4 并发 `complete()` | **1.01s → 并行**（网关无罪） |
| 真实 graph 扇出（central） | sleep 替身 LLM + 真 retrieve（qdrant 关） | 6 资源 gen 间隔 2s **串行**，total 18.59s |
| 同 ↑ 但 retrieve 替身置空 | | 6 gen 同刻启动 span=0.00s **并行**，total 4.34s |

**根因坐实**：串行点在 `generate_resource` 内的 **RAG retrieve**——`semantic.embed_query` / `rerank.predict` 是**同步 CPU**，直接在事件循环上跑会阻塞，使并发的 N 个资源 retrieve 逐个串行（其 LLM 调用因此无法重叠）；外加 `KeywordIndex` 构建失败**未缓存**，qdrant 不可用时每个资源各自重试慢构建（在 `_build_lock` 上排队）。

**修复（守 OOM/双模型不并发铁律）**：
- 新增 `rag/model_infer.run_model(fn,...)`：`asyncio.to_thread` 把同步推理丢线程（**事件循环让出 → LLM 调用得以重叠**）+ 全局 `threading.Lock` 串行实际推理（**绝不让 embedding/reranker 并发**，loop 无关）。`semantic.embed_query`、`service._rerank_or_fallback` 接入。
- `KeywordIndex`：构建失败进 30s 冷却（窗口内 `get()` 直返 None 不重试，collapse N 资源的串行重试风暴）；`invalidate()` 全量重置（含 `_build_lock`，修复跨事件循环复用锁的潜在坑）。

**验证**：修复后同实验 6 资源 gen span=0.00s **并行**，total 18.59s→8.38s（余 7s 为测试环境 qdrant/redis 关的 retrieve 超时前奏，非扇出）。RAG +5 测试，全量 **594 passed**。

**诚实边界**：扇出并行性已在真 graph 坐实；**端到端 TTFC/total 实降仍需真实可用中转站活体采集**（当前 relay 502）。投影：working relay 下 6 资源 `total 6×(retrieve+LLM) 串行 → ≈max 单资源链`（docs §3 目标 <35s）。前置串行链（profile→planner→…→path_plan 仍 4–5 次串行 LLM）属 **PERF-B**（合并前置/cheap 路由）范畴，未本轮处理。

## 8. PERF-B 进度（2026-06-15 · cheap 路由半）

PERF-B = 「砍串行前置」+「评判类任务 cheap 路由」。本轮做**后半（cheap 路由）**，前半（merge profile+planner）见下「待办」。

**做了什么**：
- `CHEAP_TASK_TYPES` 扩为 `{summary, verification, judgment, reasoning}`——质检/judge/反思(critic·metacognition·debate) 走便宜档；**不含 generation/planning/profiling**（定核心产出与个性化，不降档）。
- **修复 cheap 路由对中转站失效的旧坑**：原 `_select_model` 对 openai_compat 一律返回主模型，且 cheap→`summary_model` 在中转站下会 key 查找失败。新增 `openai_compat_cheap_model` 配置；中转站下评判任务在配了便宜档时走它（裸名进 wire payload、复用中转站 key+base_url），**默认空=全用主模型零回归**。
- provider 档（qwen/anthropic）：评判任务默认走便宜档（qwen-turbo/haiku），与既有 summary 便宜档一致的设计哲学。

**验证**：gateway +4 测试（评判任务 cheap、中转站便宜档路由+key/base、默认零回归、wire payload model 字段），全量 **598 passed**。

**诚实边界 + 待办（PERF-B 前半「砍串行前置」未做）**：
- 中转站生产路径**默认零变化**——须运维配 `openai_compat_cheap_model` 才生效（需知道中转站有哪些便宜模型 + eval ablation 守门验质量不回归）。本轮只把能力做对做全，未替运维选型。

- 前置串行链现状仍为 profile→planner→…→path_plan 4–5 次串行 LLM；cheap 路由降的是评判类**单次时延**，不减**次数**。

## 9. PERF-C 进度（2026-06-15 · 连接复用 + PERSIST 后台化，FS-14）

超时分级已于 FS-8 完成（`llm_connect_timeout_s` 独立 read）。本轮补齐另两项：

**① 共享 httpx client（连接池复用）**：新增 `llm_gateway/http_client.py` 的模块级 `AsyncClient` 单例（`get_async_client`/`reset_async_client`/`aclose_async_client`）；`gateway._complete_openai_compat` 与 `streaming._stream_openai_compat` 改用它，省每次 LLM 外呼的 TCP+TLS 握手（docs §1 因素②，~0.3–1s/次 × 每请求 5–15 次）。timeout 沿用 read/connect 分级。放独立模块避 gateway↔streaming 循环导入。应用 lifespan 退出时 `aclose`。

**② PERSIST 后台化**：`run_session` 的写回 Redis / 递归摘要 LLM / promote / autogrow 从「yield 耗尽后同步执行」改为 `asyncio.ensure_future` 后台 task（`_spawn_persist` + `_PERSIST_TASKS` 强引用防 GC）。**done 帧不再等待**这些（多轮溢出递归摘要 LLM 时省 3–10s）。`drain_persist_tasks()` 供单测断言副作用前收尾 + lifespan 优雅关停。`_persist_session` 抽为独立函数（异常吞掉记日志，降级铁律不变）。

**验证**：gateway client 复用 +1、PERSIST 后台化 +1 测试（persist 卡 gate 时 run_session 消费已结束 → 证明不阻塞）；3 处 run_session 测试消费方加 `drain_persist_tasks()`。**全量 600 passed**。

**诚实边界**：连接池/后台化的端到端时长收益仍需真实可用中转站活体采集（relay 502）。连接池省的是握手（每次几百 ms），PERSIST 后台化省的是 done 帧等待（仅多轮溢出时显著）。
