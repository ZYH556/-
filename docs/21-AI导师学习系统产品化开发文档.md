# 21 · AI 导师学习系统产品化开发文档

更新时间：2026-06-12  
文档性质：正式开发文档  
适用范围：登录后学习系统、Today 首页、AI 导师、学习画像、资源生成、学习路径与测试数据建设

---

## 1. 本轮结论

ReflexLearn 的产品方向应回到最初设想：

> 一个接入 AI 的 1 对 1 学习导师系统。它通过对话理解学生，动态构建学习画像，组织多 Agent 协作生成资源，并把资源、错题、路径和成长档案连接成持续进化的学习闭环。

当前项目已经具备多智能体、RAG、资源生成、错题复盘、协作轨迹、知识库、资源库、登录注册等技术底座。2026-06-12 已完成第一轮登录后产品化：Today 首页、`/api/today` 聚合接口、学习空间详情、资源平台来源表达、AI 导师页和演示 seed 数据已形成 MVP。

下一阶段主线不是继续堆页面，而是做产品闭环验收、AI 学伴接入、画像/路径/成长档案深化，并把演示链路整理成可稳定展示的学习系统。

## 2. 为什么之前设计跑偏

### 2.1 公开首页和登录后首页被混在一起

前一阶段多次讨论“首页”，实际存在两个不同对象：

- 未登录公开首页：用于品牌、产品定位和登录入口。
- 登录后首页：用于学习推进、任务调度和用户行动。

此前改动主要围绕公开首页和登录页视觉展开，导致登录后的学习系统首页没有被真正设计。

### 2.2 技术能力被直接映射成页面菜单

当前左侧导航包括学习空间、对话、路径、资源库、知识库、错题本、成长档案。这些能力本身是必要的，但它们只是系统模块，不等于用户工作流。

用户进入系统时并不关心“有哪些模块”，而是关心：

- 我当前学到哪里了？
- 系统建议我接下来做什么？
- 我现在点哪里可以开始？
- 这些资源和错题如何帮我进步？

因此登录后首页必须从“模块目录”转为“学习驾驶舱”。

### 2.3 AI 导师的角色没有被产品化

项目已有 Chat / Agent / Profile / Planner / Generator / Critic 等技术链路，但前端表达仍像一个通用聊天工作区。真正的产品核心应是：

- AI 导师主动询问学习情况。
- 对话用于构建学习画像，而不是只回答问题。
- 多 Agent 协作隐藏在资源生成和路径规划背后。
- 用户能看到“系统记住了什么、为什么推荐这一步”。

### 2.4 数据和资源不足导致页面空壳感

虽然系统已有学习空间、资源库、错题本等页面，但缺少足够真实的示例数据。没有学生画像、课程资源、错题、路径和资源样本时，页面只能呈现空状态或列表壳。

下一阶段必须准备真实结构的测试数据，包括学生画像样本和 100+ 学习资源，让产品体验先成形。

## 3. 赛题痛点与产品目标

### 3.1 需要解决的痛点

1. 不同专业、不同学历水平的学生面对海量课程资料、学术文献和学习工具，难以快速筛选适合自身进度和能力的资源。
2. 课堂集体讲授无法兼顾每位学生的节奏和特点，导致知识掌握和能力提升存在明显差距。
3. 传统学习模式和基础智能辅助系统缺少多模态生成、多智能体协同等能力，难以满足现代高等教育对创新型、个性化人才培养的要求。

### 3.2 产品目标

ReflexLearn 要提供：

- 对话式学习画像自主构建。
- 至少 6 个维度的动态学生画像。
- 至少 5 类个性化学习资源生成。
- 多 Agent 协作资源生成与质量校验。
- 基于画像、进度、掌握情况、学习偏好的动态路径规划。
- 学习过程中随时可用的微型智能辅导。
- 资源、错题、路径、成长档案的闭环管理。

## 4. 目标用户体验

### 4.1 用户进入系统后的第一感受

用户登录后进入 **Today / 今日学习**。

它不展示后台框架，而是直接回答：

- 今天建议学什么？
- 为什么建议这一步？
- 当前薄弱点是什么？
- 继续学习、问导师、上传资料、录入错题分别从哪里开始？

### 4.2 AI 导师的定位

AI 导师不是客服，也不是通用聊天机器人。它是一个 1 对 1 学习导师：

- 会问学生学习目标、专业背景、当前基础和时间安排。
- 会记住学习进度和偏好。
- 会基于错题、资源、路径、对话持续更新画像。
- 会在学习过程中提供即时答疑、图解、短视频讲解、代码案例和补救建议。

### 4.3 AI 学伴小人

系统不采用普通的全局悬浮按钮。登录后的学习系统需要一个可移动、可反馈状态的 AI 学伴小人，作为随身导师入口。

它应具备页面上下文：

- 当前用户是谁。
- 正在看哪个学习空间。
- 当前学习目标是什么。
- 当前页面对象是什么资源、错题或路径节点。
- 后台是否正在检索、生成资源、分析错题或调整路径。

它支持：

- 点击打开微型对话框。
- 文字答疑和图解说明。
- 请求生成短视频/动画脚本。
- 生成代码实操案例。
- 将解释结果加入资源库。
- 将当前问题写入错题本或学习计划。
- 在后台任务运行、完成、失败、需要选择时通过动作和状态变化反馈给用户。

第一版可用 CSS/SVG 小人实现移动、眨眼、思考、工作、提示等状态，后续再升级为精灵图或更完整的桌面宠物式动画。它的定位是“学习陪伴代理”，不是装饰物。

## 6. 当前未完成或不能夸大的能力

### 6.1 产品体验缺口

- 学习画像没有独立页面和用户可读展示。
- 学习路径页面仍偏说明，尚未形成完整路径管理体验。
- 成长档案偏协作轨迹，不像学生成长面板。
- 资源库仍缺资源详情、收藏、版本、下载和站内嵌入播放详情页。
- 页面之间已形成 Today -> 学习空间 -> 资源库 / AI 导师 / 错题复盘 的初步连接，但“目标 -> 路径 -> 资源 -> 错题 -> 成长”的闭环还需要继续深化。
- AI 学伴小人仍在独立实现中，不能用普通浮窗按钮替代。

### 6.2 数据缺口

- 已有第一批 seed 画像、空间、资源、错题样本，但还未扩充到 100+ 可演示学习资源。
- 缺课程数据、章节数据、讨论样例。
- 缺差距报告样本。
- 缺导师样本。
- 学习路径样例仍需扩充，尤其要覆盖不同专业、不同学历、不同目标的路径差异。

### 6.3 技术与上线缺口

- Google/GitHub 不是真 OAuth。
- SeeDance、D-ID、HeyGen、Manim 等多模态生成还不是完整生产链路。
- LoRA 还没有完成训练和收益证明。
- Safety 仍是基础规则版。
- 外部依赖真实服务矩阵下的稳定 CI 证据不足。

## 8. Today / 今日学习设计

### 8.1 页面定位

Today 是登录后默认首页。它是学习驾驶舱，不是功能目录。

### 8.2 第一屏结构

推荐结构：

```text
顶部：问候 + 当前主目标 + 画像摘要

左列：继续学习
中列：下一步建议
右列：快速入口 / AI 导师入口

下方：最近资源 / 错题提醒 / 成长变化
```

### 8.3 关键模块

#### 当前学习目标

展示：

- 主学习空间。
- 目标名称。
- 学习阶段。
- 最近一次学习位置。

#### 下一步建议

首页视觉权重最高的模块。

内容：

- 推荐动作。
- 推荐理由。
- 预计耗时。
- 对应入口。

示例：

> 建议先复习线性回归损失函数，再完成 5 道梯度下降练习。因为你的错题集中在“代价函数”和“参数更新”。

#### 快速入口

入口必须是动作，而不是菜单：

- 和 AI 导师聊聊。
- 创建学习目标。
- 上传课程资料。
- 录入一道错题。
- 生成学习路径。
- 生成一组练习。

#### 最近动态

展示系统正在学习用户：

- 画像更新。
- 新资源生成。
- 错题复习提醒。
- Agent 调整了路径。

## 9. AI 导师设计

### 9.1 主对话页

AI 导师页应类似 ChatGPT，但要强化学习场景。

首屏空状态应提供导师式引导：

- 你正在准备什么课程、考试或项目？
- 你目前的基础如何？
- 你希望系统先帮你规划路径、解释知识点，还是生成资源？

### 9.2 对话目标

对话不只是问答，还要产出结构化结果：

- 学习画像。
- 学习目标。
- 学习路径。
- 资源生成任务。
- 错题分析。
- 下一步建议。

### 9.3 AI 学伴小人

不采用普通的全局悬浮按钮。登录后的学习系统需要设计一个可移动、可反馈状态的 **AI 学伴小人**，作为随身导师入口。

它不是装饰物，而是学习系统的交互代理：

- 空闲时在页面边缘轻微移动或停留。
- 用户学习时保持低干扰状态。
- 后台生成资源、检索资料、分析错题时，小人进入工作状态。
- 出错或服务降级时，小人出现疲惫/提示状态。
- 有新建议、错题提醒、路径更新时，小人主动提示。
- 点击后打开上下文对话框。

小人状态建议：

| 状态 | 含义 | UI 表现 |
| --- | --- | --- |
| idle | 空闲陪伴 | 轻微呼吸、眨眼 |
| thinking | 正在思考 | 看书、转笔、头顶小进度 |
| running | 正在执行任务 | 走动、敲键盘、翻资料 |
| waiting | 需要用户选择 | 举手、看向用户 |
| success | 已生成建议或资源 | 开心提示 |
| failed | 服务失败或降级 | 低落、提示重试 |
| review | 正在检查资源质量 | 放大镜/审阅姿态 |

第一版不需要完整动画精灵图，可以先用 CSS/SVG/简单序列帧实现，但组件结构要为后续动画 atlas 留出口。后续如需更完整的桌面宠物体验，可参考 Codex pet 的状态机方式，制作统一的 sprite sheet。

### 9.4 小人打开的辅导框

AI 学伴小人点击后打开微型辅导框，后续出现在：

- Today。
- 学习路径。
- 资源详情。
- 知识库文档。
- 错题详情。
- 课程播放页。

第一版可只接入当前页面上下文和普通对话；后续再支持多模态解答。

辅导框必须携带上下文：

- 当前页面。
- 当前学习空间。
- 当前资源、错题或路径节点。
- 当前学生画像摘要。
- 最近一次系统建议。

这能避免它变成普通客服窗口。

## 10. 学习画像设计

### 10.1 画像维度

至少 6 个维度：

| 维度 | 示例 |
| --- | --- |
| 专业方向 | 计算机、人工智能、数学、自动化 |
| 学历阶段 | 大一、大二、大三、大四、研究生 |
| 知识基础 | 掌握/薄弱知识点 |
| 学习目标 | 考试、求职、课程补弱、竞赛、科研 |
| 学习进度 | 当前路径节点、完成率、最近学习时间 |
| 学习偏好 | 文档、视频、练习、代码实操、图解 |
| 错题模式 | 常错知识点、错误原因 |
| 能力短板 | 计算、理解、应用、表达、工程实践 |

### 10.2 随学随新

画像更新来源：

- AI 导师对话。
- 学习路径完成情况。
- 错题本。
- 资源使用偏好。
- 知识库上传内容。
- 课程学习行为。

第一版允许先用画像摘要占位，但 UI 和数据结构要为真实画像 API 留出口。

## 11. 多 Agent 分工

至少设计以下专业 Agent：

| Agent | 职责 | 对应用户价值 |
| --- | --- | --- |
| 画像 Agent | 抽取和更新学生画像 | 系统理解我 |
| 路径规划 Agent | 生成学习路径和顺序 | 我知道下一步 |
| 检索 Agent | 从课程、知识库、文献召回内容 | 找到适合我的资料 |
| 文档 Agent | 生成课程讲解文档 | 我能系统理解知识 |
| 导图 Agent | 生成知识结构图 | 我能看到知识关系 |
| 题库 Agent | 生成选择、填空、简答、编程题 | 我能练习和检测 |
| 阅读 Agent | 推荐拓展阅读 | 我能深入学习 |
| 视频 Agent | 生成教学视频/动画分镜 | 我能多模态理解 |
| 代码实操 Agent | 生成代码案例和项目练习 | 我能动手实践 |
| 评估 Agent | 评价资源质量和画像匹配 | 内容更可靠 |
| 元认知 Agent | 分析错因和改进策略 | 系统越用越懂我 |

前端不应把 Agent 机械摆成一排图标。它们应隐藏在学习流程中，只在资源生成、路径规划、协作轨迹等场景展示过程。

## 12. 个性化资源生成

赛题要求至少 5 种资源，项目应覆盖 6 种：

| 资源类型 | 当前状态 | 下一步 |
| --- | --- | --- |
| 专业课程讲解文档 | 已有生成链路 | 增强结构和引用来源 |
| 知识点思维导图 | 已有生成链路 | 接 Markmap 预览和导出 |

资源生成必须受画像约束：

- 不同专业给不同例子。
- 不同学历阶段给不同深度。
- 不同掌握情况给不同难度。
- 不同偏好给不同资源类型。

### 12.1 资源生成必须调用 Skill

资源不能凭空生成。每一种资源生成都必须通过明确的 Skill 或工具链完成，生成过程需要可追踪、可复用、可降级。

建议 Skill 分层：

| Skill | 职责 |
| --- | --- |
| `resource_search` | 根据主题搜索外部公开资源和平台链接 |
| `oer_ingest` | 拉取可开放入库的 OER 文本资源 |
| `bilibili_search` | 搜索 B 站学习视频，返回元数据和可播放链接 |
| `official_docs_search` | 检索官方文档、教材、开放课程 |
| `paper_search` | 检索论文、综述、学术资料 |
| `doc_gen` | 基于检索内容和画像生成讲解文档 |
| `mindmap_gen` | 基于讲解文档生成思维导图 |
| `quiz_gen` | 生成不同题型和解析 |
| `reading_gen` | 生成拓展阅读清单和摘要 |
| `video_gen` | 生成教学视频脚本、分镜或外部视频推荐 |
| `code_gen` | 生成代码实操案例 |
| `quality_check` | 校验资源质量、难度和画像匹配度 |

每个资源卡片应记录：

- 来源。
- 使用的 Skill。
- 是否 AI 生成。
- 是否可入库全文。
- 是否只是外部链接推荐。
- 生成时使用的画像摘要。
- 质量检查结果。

### 12.2 外部资源策略

资源来源分三类：

| 类型 | 处理方式 | 示例 |
| --- | --- | --- |
| 开放授权资源 | 可入库全文，进入 RAG | MIT OCW、OpenStax、OER Commons、官方文档 |
| 平台学习资源 | 只存标题、链接、封面、作者、标签，不抓视频正文 | B 站、Coursera、edX、中国大学 MOOC、学堂在线 |
| AI 个性化资源 | 存全文，归入个人资源库 | 文档、导图、题目、视频脚本、代码案例 |

关键原则：

- 不把外部平台内容伪装成自有资源。
- 不抓取 B 站、Coursera、MOOC 等平台的视频和课程全文。
- 可以做搜索、推荐、嵌入播放、跳转观看。
- AI 生成资源必须标注为 AI 生成，并注明参考了哪些来源。

第一阶段建议优先接入以下来源：

| 来源 | 用途 | 入库策略 |
| --- | --- | --- |
| B 站 | 中文免费视频推荐、站内嵌入观看 | 保存元数据、BV 号、封面、链接和 iframe 地址 |
| OpenStax | 英文开放教材、基础学科内容 | 按授权保存章节元数据，允许时拉取正文进入 RAG |
| MIT OpenCourseWare | 公开课程、讲义、作业 | 保存课程、讲义、视频链接和授权信息 |
| 官方文档 | 编程、框架、工具链学习 | 保存链接、标题、摘要，必要时按许可抓取正文 |
| arXiv / Semantic Scholar | 论文和综述 | 保存题名、作者、摘要、链接、引用信息 |
| 用户上传资料 | 课程 PPT、笔记、作业要求 | 归入个人知识库，受权限控制 |

推荐资源平台要同时支持“外部观看”和“个人学习资源”两种形态。外部观看负责带学生找到好内容，个人学习资源负责承载 AI 导师生成的讲解、导图、题目、代码案例和补救计划。

### 12.3 B 站学习视频接入

B 站适合成为中文学习资源推荐入口。很多学生确实会通过 B 站学习，因此系统应支持：

1. 根据学习目标、知识点和画像生成搜索词。
2. 调用 B 站搜索能力或安全的网页搜索方式获取视频候选。
3. 保存视频元数据：标题、UP 主、BV 号、链接、封面、时长、播放量、标签。
4. 根据画像和当前路径排序推荐。
5. 使用 B 站外链播放器嵌入观看。
6. 不下载、不转存、不抽取视频内容作为自有数据。

B 站官方提供站外播放器，引用地址为 `https://player.bilibili.com/player.html`，可使用 `bvid` 等参数嵌入播放。搜索接口应优先走公开合规能力，避免依赖非公开 API。

资源卡片示例字段：

```text
resource_type: external_video
provider: bilibili
title: 线性回归通俗讲解
url: https://www.bilibili.com/video/...
embed_url: https://player.bilibili.com/player.html?bvid=...
usage_mode: metadata_only
license: platform_link
source_policy: embed_or_redirect_only
```

### 12.4 总学习资源平台

系统需要一个统一的“学习资源平台”，不是简单资源库。

它应整合：

- 外部开放资源。
- B 站等免费视频资源链接。
- 官方文档和论文。
- AI 生成资源。
- 用户上传资料。
- 错题补救资源。

核心能力：

- 搜索。
- 筛选。
- 按画像推荐。
- 按学习路径挂载。
- 区分“外部观看”和“个人资源”。
- 支持一键让 AI 导师解释某个资源。
