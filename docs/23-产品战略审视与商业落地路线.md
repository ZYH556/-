## 1. 主线校验：有没有跑偏

**双主线**：①软件杯 A3「自进化学习多智能体系统」比赛交付；②长期「自进化学习平台」产品化（docs/14）。

**结论：没有脱离，但存在一个需要警惕的张力。**

- 当前所有工作（画像快照历史、资源发现、档案化视觉、保存闭环）都落在 docs/21 产品化文档划定的「学习闭环」主线上，且每项都直接服务比赛的「创新性 + 完成度」评分项。
- **张力点：演示导向 vs 真实可用。** 现在系统的「好看程度」已经领先「真实数据厚度」：seed 数据撑起了 UI，但真实用户从零开始时（无画像/无错题/无资源）的冷启动体验，与演示账号差距很大。这不是跑偏，而是阶段必然——但从本轮起，每做一个新面，必须同时回答「空状态长什么样、第一次使用怎么引导」。目前各页空状态已有兜底文案，引导动作（CTA）已接，可以接受；真正薄弱的是「第一次对话 → 第一份画像 → 第一条路径」的新手链路没有专门设计。

## 2. 当前实现效果（截至本轮）

**已经成立的产品事实**（活体验证过，非 PPT 描述）：
1. **学习闭环骨架**：对话生成资源/路径 → 自动沉淀学习空间 → Today 驾驶舱聚合 → 错题飞轮（归因/补救/资源/复习回写）→ 画像快照 → 成长趋势。每一段都有 API + 前端 + 测试。
2. **画像快照有真实时间维度**（本轮修复后）：内容变化才写快照，趋势窗口不再被同质快照灌满；/growth 有不足 2 份快照的诚实占位。
3. **候选资源「发现 → 保存 → 资源库」闭环**（本轮新增）：discover 候选带 metadata_only 红线，保存幂等（candidate_id），落 PG resources 表进入既有 ACL/列表体系。
4. **统一设计语言**：「学习档案」印刷隐喻铺满 /profile /plan /growth（印章、铅字序号、刻度尺、火花线、对账单 leader dots），有别于通用 AI 产品的紫色渐变审美，构成可辨识的品牌面。
5. **工程底座**：500+ hermetic 单测、降级矩阵纪律、cookie+CSRF+限流+审计+上传隔离的安全包、Prometheus 指标。

## 3. 多视角审视：不足与风险

### 3.1 真实学习者视角（最大的产品风险）
- **资源是「方向」不是「内容」**：discover 返回的是搜索链接模板，不是真实视频/文档条目。用户点「打开来源」跳到 B 站搜索页——这在演示里成立，真实使用中价值有限。→ 路线 §4-W2 真实搜索。
- **学习行为没有回流**：保存了资源、看完了视频、做完了练习，系统不知道「看完没有、有没有用」。画像 progress 长期为 0 的根因是没有行为事件（完成/复盘/用时）回写。→ 这是比「更多页面」更重要的下一块拼图（§4-W3 资源详情页 + 学习状态）。
- **AI 导师等待体验**：真实 LLM 全链路 ~75s，无中间态时用户会认为挂了。→ §4-W2 中间态。

### 3.2 商业视角
- **成本结构**：一次完整会话 5 次串行 LLM 往返（docs/19），token 成本与时延双高。商业化前必须做 PERF 序列（缓存、并行、流式、小模型路由已有地基）。
- **差异化壁垒**：当前壁垒是「闭环数据模型 +自进化机制」，不是单点功能——错题→画像→路径→资源的数据飞轮一旦有真实用户数据，迁移成本就是护城河。所以「行为回流」也是商业优先级。
- **付费点假设**（待验证）：个性化路径 + 错题补救（C 端订阅）；班级/机构视角的画像看板（B 端）。多租户底座已在，B 端是低改造成本的方向。

### 3.3 压力测试 / 上线视角
- **并发**：asyncio 后台任务（视频/沉淀）在单进程内，多 worker 部署时 JobStore 靠 Redis 但任务执行不跨进程恢复；限流进程内存版多实例不共享（已知，生产需 Redis 强制）。
- **数据一致性**：资源写侧三处入口（chat 沉淀 / 上传 / 候选保存）契约靠约定（meta JSONB 键名），无 schema 校验层——字段漂移风险。建议后续把 meta 键收敛成 Pydantic 模型统一序列化（轻改造）。
- **外链合规**：B 站 embed/跳转遵守 metadata_only + embed_or_redirect_only 红线已是正确姿势；上线前需补 robots/ToS 复查与「内容不可用」兜底（外链 404 检测）。
- **OAuth/生产化**：Google/GitHub OAuth、HTTPS、反代同源、日志轮转、备份恢复未配置——上线级检查单见 §4-W4。

### 3.4 比赛评审视角
- 消融证据链已可跑（real_full vs no_rag vs single_agent），但样本量小、缺人工抽检表；LoRA 仍是空白（可选项）。
- 演示链路（docs/22）需要把本轮新面（趋势/保存闭环）编入演示脚本，体现「自进化」叙事：**画像会变 → 路径会变 → 证据可查**。

## 4. 路线编排（融合用户 9 方向 + 本轮审视）

> 原则：每波都「先闭环后扩面」，能落库的优先于能展示的。✅=本轮已完成。

**W1 · 沉淀闭环补完（最高优先）**
- ✅ 候选资源保存（幂等入库 → 资源库/列表）
- ✅ 画像快照去重（内容变化才写；同天节流可后续再加）
- ✅ 资源详情页 `/resources/[id]`：来源/理由/关联目标/学习状态四档（unread/in_progress/done/reviewed）点选即存
- ✅ 行为回流最小闭环：`study_stats` 进画像聚合与快照 payload（状态变化驱动新快照），/growth 学习证据卡显示资源使用效果与错题复盘率；画像 progress 公式重算待行为数据积累后定口径

**W2 · 真实数据与体验**
- ✅ 真实搜索接入（B 站）：spike 结论——公开搜索接口需 wbi 签名（nav 接口取 img/sub key → 固定置换表 mixin → 参数排序 MD5）+ buvid3 cookie，已实现 `learning/bilibili_search.py`（签名/关键词缓存 1h/滑窗限频 10 次每分钟/`trust_env=False` 防死代理/任何失败返 None 回落静态候选）；搜索词用 goal+薄弱点组合（单独薄弱点太泛会跑题）；conftest 类级守卫禁单测真连。维护风险：wbi 算法 B 站可能调整，挂了自动降级静态候选不影响主链路。官方文档/公开课程搜索仍为静态候选（后续逐源接入）。
- ✅ AI 导师中间态：牛牛面板 `CompanionThinking` 时间驱动四阶段文案；/chat 主对话本有 agent_step 流。
- ✅ 数据扩充第一步：`scripts/seed_product.sh` 接线，24 条带 provider 种子入库；继续扩到 100+ 真实方向数据待真实搜索保存沉淀。

**W3 · 路径管理与个性化深化**
- /plan 节点操作：标完成/插入补救节点（错题飞轮已有 plan 生成，差「插入现有路径」）；候选资源绑定路径节点
- /growth 深化：资源使用效果、错题复盘完成率（依赖 W1 行为事件）
- 牛牛面板避让 + 状态联动打磨

**W4 · 上线级工程**
- PERF-B→C→A→D（docs/19：并行化/缓存/流式/小模型路由）
- OAuth（Google/GitHub）+ 生产 cookie 配置复查
- 部署清单：HTTPS/反代/日志轮转/备份/监控告警阈值/外链健康检查
- 压测脚本（locust 或 k6）：聊天 SSE 并发 + 资源列表 + 保存幂等竞态

**资源缺口（按用户指示跳过待补）**：LoRA GPU、SeeDance key、真实搜索 API 配额、OAuth 应用审核。机制全部降级占位先行。
